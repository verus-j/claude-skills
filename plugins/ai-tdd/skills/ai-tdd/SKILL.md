# AI-TDD 협업 워크플로우

## 핵심 철학

**테스트는 AI를 통제하는 계약서다.** AI는 그럴듯하지만 틀린 코드, 또는 요구하지 않은 코드를 쓰는 경향이 있다. 이를 막는 장치는 ① 먼저 실패하는 테스트(계약), ② 사람의 조향(승인)이다. 목표는 AI의 판단을 없애는 게 아니라 **보이게 만드는 것**.

## 전체 흐름

```
합의 단계 (계획 모드에서 시작, 1~3차 승인)
  → 합의문·리스트를 두 파일(testlist, refactorlist)로 저장 + 합의 커밋
  → [사이클: Red → Green → Refactor → 보고·다음 작업 선택 → 사이클 커밋] 반복
  → 완주 후 커버리지 점검
```

## 역할 분담

- **AI**: 테스트·구현 작성, 사이클 실행, 설계 트레이드오프 분석과 추천, Red/Green 예측(틀리면 정정)
- **사람**: 각 합의의 승인, 다음 작업 선택, 리팩토링 방향 조향, 스펙 결정, 요구사항 페이스 조절

## 핵심 규칙 6가지

1. **한 번에 실패하는 테스트 하나** — 미래 요구사항을 앞서 구현하지 않는다
2. **Red를 눈으로 확인** — 구현 전에 반드시 테스트를 실행해 실패 출력을 증거로 공유한다
3. **테스트를 통과시키는 최소 구현만** — 하드코딩 허용. 일반화는 다음 테스트가 강제할 때
4. **AI는 테스트를 임의로 수정하지 않는다** — 테스트 변경은 반드시 사람 승인
5. **Refactor는 Green에서만** — 무엇을 왜 바꾸는지 설명하고 진행하며, 행동을 보존한다
6. **사람의 역할 = 조향** — 설계 결정이 갈리는 지점에서는 트레이드오프를 제시하고 사람이 결정한다

## 합의 단계 (코드 작성 전 필수, 계획 모드에서 시작)

합의 단계는 계획 모드에 진입해 시작한다. 코드 수정이 차단된 상태에서 컨텍스트를 쌓는다. 사이클이 계약의 이행이라면, 합의 단계는 계약을 쓰는 시간이다.

**승인은 한 번이 아니라 단계별로 받는다** — 도메인 합의가 확정되기 전에 리스트까지 한 뭉치로 승인받으면 승인의 해상도가 떨어진다:

1. **1차 승인 (계획 모드 승인)**: 합의 1(기능 정의·도메인 규칙)과 합의 2(API·구현 방향 스케치)까지만 계획으로 제시하고 승인받는다. 테스트 리스트는 계획에 포함하지 않는다
2. **2차 승인**: 확정된 합의 1·2 위에서 합의 3(테스트 리스트)을 작성해 별도로 제시하고 승인받는다
3. **3차 승인**: 합의 4(진행 모드)를 별도로 논의해 확정한다

### 합의 1. 기능 정의와 도메인 규칙 정리

- 만들 기능이 무엇인지 한두 문장으로 정의하고, 구현에 필요한 도메인 규칙만 짧게 요약한다
- 용어집을 정리한다 — 여기서 확정한 어휘가 코드·테스트의 이름이 된다
- 대표 시나리오의 기대값을 손으로 계산해 예제 표로 정리한다 (테스트 오라클 검증 — 기대값이 틀린 테스트는 TDD 전체를 오염시킨다)
- **암묵 스펙을 표면화한다**: 테스트 리스트의 모양 자체가 스펙 결정을 내포할 수 있다 (예: "프레임 하나짜리 테스트 허용" = "진행 중 게임의 score() 조회 허용"이라는 스펙 채택). 이런 결정은 선택지+추천으로 제시하고 합의한다
- 정의되지 않은 행동(예: 게임 완료 후 추가 투구)을 발견하면 리스트에 넣을지 스코프 밖으로 선언할지 결정한다

### 합의 2. 공개 API 합의와 구현 방향 스케치

- 테스트가 사용할 인터페이스를 먼저 합의한다 (예: `game.roll(pins)` / `game.score()`). 도메인 개념(예: 프레임)을 API에 노출할지 내부에 둘지도 이 시점에 언급한다
- **구현 방향을 가볍게 스케치한다**: 유력한 내부 구조 후보들을 트레이드오프와 함께 한두 문장씩 논의한다 (예: 투구 리스트 + 인덱스 순회 vs 프레임 객체 + 즉시 정산). 단, 스케치는 **약속이 아니라 가설이다** — 테스트가 다른 방향을 강제하면 폐기하고, 스케치를 근거로 선행 구현하지 않는다 (규칙 1은 그대로 유효)
- API와 구현 방향이 먼저 정리되어야 다음 단계의 리스트 품질 점검(간격, 공짜 통과 예측)이 정확해진다

### 합의 3. 테스트 리스트 작성

각 항목은 **관찰 가능한 행동 하나 + 대표 입출력 예시**로 쓴다 (예: "스페어 보너스 — 5,5 / 3 → 16"). 예시가 있어야 사이클 중 "공짜 통과" 판단과 Red 예측이 정확해진다. **항목은 공개 API로 관찰 가능한 행동만 다룬다** — 합의 2에서 스케치한 내부 구조를 검증하는 항목을 넣지 않는다 (리스트는 행동의 계약이지 설계의 계약이 아니다).

리스트 품질 점검 4가지:

1. **크기**: 첫 테스트가 전체 시나리오를 요구하면 더 작은 계단으로 분할한다 (전체 게임 → 프레임 하나)
2. **커버리지**: 특수 규칙마다 ① 단독 케이스 ② 조합·연속 케이스 ③ 경계 케이스가 각각 있는지 점검한다. 통합형 테스트(예: 퍼펙트 게임)에만 의존하면 단독 케이스가 그 안에 숨어 실패 신호가 뭉개진다
3. **간격**: 인접 항목 사이의 구현 점프가 너무 크면 중간 계단을 추가한다 (단일 스트라이크 → 퍼펙트 게임 사이에 연속 스트라이크)
4. **정렬**: 뼈대(단순 경로) → 특수 규칙 → 조합 → 경계·통합 순으로 배치한다

**문서화 테스트 후보를 미리 표시한다**: 앞 항목의 자연스러운 구현이 통과시킬 것으로 예상되는 항목(예: 거터 게임)은 리스트에 표시해둔다. 예측이 맞으면 문서화 테스트로 추가하고, 틀리면 정상 사이클로 진행한다 — 어느 쪽이든 데이터가 된다.

### 합의 4. 진행 모드 합의

기본은 사이클 단위 승인 — 사이클이 끝날 때마다 요약을 보고하고 다음 작업 선택을 기다린다. git 사용 여부와 커밋 규칙 적용도 이때 확정한다.

### 합의 완료 후: 리스트 파일이 단일 진실 공급원

- 1~3차 승인이 모두 끝나면 첫 행동으로 두 파일을 만들고 합의 커밋을 남긴다:
  - `docs/tdd/<주제>-testlist.md` — 합의 내용 전체 (도메인 규칙 요약, 예제 표, API, 구현 방향 스케치, 테스트 리스트, 진행 모드)
  - `docs/tdd/<주제>-refactorlist.md` — 리팩토링 리스트 (시작 시 빈 목록). 보류된 리팩토링 후보를 보류 사유와 함께 기록하고, 완료 시 체크한다
- 이후 사이클마다 두 파일의 체크박스와 문서화 테스트 표시를 갱신한다 — 대화가 아니라 파일이 리스트의 단일 진실 공급원이다
- **완료된 항목도 파일에서 삭제하지 않는다** — 체크 표시(`[x]`)로 유지해 전체 이력을 보존한다. 남은 항목만 보여주는 것은 사이클 종료 보고의 표에서만이다
- 리스트는 살아있는 문서다 — 진행 중 케이스 추가·분할·수정이 가능하며, 변경은 사람이 승인한다 (아래 "진행 중 새 테스트 케이스 발견" 참조)
- **완주 후 커버리지 점검**: 리스트를 모두 완료하면 스위트의 구멍(통합 테스트에만 숨어 있는 단독 케이스, 정의되지 않은 행동, 미검증 경계 조합)을 점검해 보고한다

## 사이클 프로토콜

### 🔴 Red

- 실패하는 테스트 하나를 작성한다 (규칙 1)
- 반드시 실행해서 실패 출력을 공유한다 (규칙 2). 예상한 이유로 실패하는지 확인한다
- 컴파일용 껍데기가 필요하면 `UnsupportedOperationException`을 던지게 만들어 assertion 단계의 실패를 관찰한다

### 🟢 Green

- 구현 전에 방침을 한 줄로 선언한다: **`[최소]`** = 이 테스트만 통과하는 구현, **`[일반화]`** = 자연스러운 최소 구현이 미래 테스트까지 통과시키는 경우. `[일반화]`는 공짜로 통과될 리스트 항목을 명시하고 진행하며, 해당 항목은 통과 확인 후 문서화 테스트로 처리한다
- 테스트를 통과시키는 최소 구현을 작성한다 (규칙 3). 하드코딩도 정답이다 — 다음 테스트가 일반화를 강제한다
- 구현 후 전체 테스트를 실행해 Green을 확인한다 (기존 테스트가 깨지지 않았는지 포함)

### 🔵 Refactor

- Green에서만 진행한다 (규칙 5). 무엇을 왜 바꾸는지 먼저 설명한다
- 행동 보존이 원칙 — **미래 요구사항 지식을 몰래 넣지 않는다**. 현재 테스트가 요구하지 않는 일반성을 도입하게 될 때는 반드시 언급한다
- 테스트는 수정하지 않는다 (규칙 4) — 리팩토링 후에도 테스트가 그대로 통과하는 것이 행동 보존의 증명이다
- 구조적 리팩토링(클래스 추출 등)은 사람이 먼저 제안할 수도 있다. AI는 찬반 트레이드오프를 모두 제시하고 추천 입장을 밝힌 뒤 조향에 따른다
- 리팩토링할 것이 없으면 "스킵"을 명시한다
- 발견했지만 지금 하지 않기로 한 후보는 **발견 즉시** 리팩토링 리스트 파일에 보류 사유와 함께 기록한다 — 사이클 중간이라도 미루지 않는다
- **다음 사이클이 바꿀 코드의 리팩토링은 보류가 기본이다.** 구조 변경은 다른 리팩토링의 비용을 바꾸므로, 보류 후보는 구조가 바뀔 때마다 재평가한다

### 보고 및 다음 작업 선택

사이클이 끝나면 아래를 보고하고, 다음 작업의 선택을 기다린다:

- 사이클 요약: Red 증거 → Green 방침·구현 내용 → Refactor 내역
- **남은 테스트 리스트와 리팩토링 리스트를 각각 별도의 표로 정리해 제시** (한 표에 섞지 않는다) — 두 표 모두 **남은 항목만** 보여준다. 리스트 파일도 함께 갱신한다. 표 형식은 아래 예시를 따른다:

  **📋 테스트 리스트 (남은 항목)**

  | # | 항목 | 예시 |
  |---|------|------|
  | 13 | 잘못된 명령 — 예외 + 위치 불변 | `FX` → `IllegalArgumentException`, `(0,0,N)` 유지 |

  **🔧 리팩토링 리스트 (남은 항목)**

  | # | 항목 | 보류 사유 / 비고 |
  |---|------|------|
  | R1 | `execute()` blocked 제어 흐름 정리 | 13번이 execute를 바꿀 예정 → 이후 재평가 |

- **다음 작업 추천**: AI는 추천(테스트 케이스 또는 리팩토링)과 근거(예상되는 Red/Green의 모양, 리팩토링이면 기대 효과)만 제시한다. **다음 작업은 사람이 선택한다** — 순서 조향도 사람의 몫이다 (규칙 6)

### 커밋 규칙 (git 사용 시)

커밋은 항상 전체 테스트 Green 상태에서만 한다. 유형별 규칙:

- **합의 커밋**: 합의 단계 산출물(두 리스트 파일) 저장 시 — `합의: <주제> 테스트 리스트와 합의 스펙`
- **사이클 커밋**: 사이클 완료(Green + Refactor 반영) 시 1커밋 — `사이클 N: <행동 요약>`. 본문에 Red 증거 한 줄을 남겨 히스토리만으로 TDD 이력을 증명하고, 테스트·리팩토링 리스트 파일 갱신을 같은 커밋에 포함한다
- **리팩토링 커밋**: 사이클 밖 구조 변경 — `리팩토링: <내용>`
- **테스트 커밋**: 구현 무변경의 테스트 스위트 변경(즉시 통과를 선언한 케이스 확장, 중복 통합·제거 등) — `테스트: <내용>`

## 특수 패턴 처리

- **진행 중 새 테스트 케이스 발견**: 구현이나 논의 중 리스트에 없는 케이스·미정의 스펙을 발견하면, 발견 즉시 구현하지 않는다 — 선택지+추천으로 스펙을 합의한 뒤 항목을 **테스트 리스트 파일에 즉시 반영**하고(발견 경위 명시), 사이클로 다룬다
- **Green without Red** (새 테스트가 즉시 통과): 원인을 진단해 보고한다. ① 과거 구현이 요구사항보다 앞서 일반화된 결과라면 그 사실을 명시하고, 테스트는 회귀 방지 가치가 있으므로 **문서화 테스트로 유지**한다. ② 경계값 등 문서화 목적으로 의도적으로 추가하는 테스트라면 추가 전에 "즉시 통과 예상"을 선언한다
- **예측과 정직한 정정**: Red/Green 모양을 예측했다가 틀리면 명시적으로 정정한다. 예측 실패는 숨길 일이 아니라 설계 이해도를 보정하는 데이터다
- **테스트 실행 증거**: 모든 Red/Green 판정은 실제 테스트 실행 출력으로 증명한다. "통과할 것이다"로 넘어가지 않는다
