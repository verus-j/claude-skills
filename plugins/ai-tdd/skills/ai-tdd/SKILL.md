# AI-TDD 협업 워크플로우

## 핵심 철학

**테스트는 AI를 통제하는 계약서다.** AI는 그럴듯하지만 틀린 코드, 또는 요구하지 않은 코드를 쓰는 경향이 있다. 이를 막는 장치는 ① 먼저 실패하는 테스트(계약), ② 사람의 조향(승인)이다. 목표는 AI의 판단을 없애는 게 아니라 **보이게 만드는 것**.

## 전체 흐름

```
합의 단계 (계획 모드에서 시작)
  → 작업 유형 분류 (동작 변경 / 기존 동작 리팩토링)
  → 1~3차 승인
  → 합의문·리스트를 세 파일(spec, testlist, refactorlist)로 저장 + 합의 커밋
  → 사이클 반복 — 한 사이클에 모자 하나:
      [RED&GREEN 사이클: Red(특성화는 Green without Red) → 최소 구현 → 전체 Green]
      [REFACTOR 사이클: 기준 Green → 행동 보존 구조 변경 → 테스트 무변경 Green]
  → 공통 후처리: 리팩토링 리스트 최신화 (직전 변경 분석 + 전체 조망)
      → 두 리스트를 확인해 다음 작업 추천 → 사람이 선택 → 사이클 커밋
  → 완주 후 커버리지 점검
```

## 역할 분담

- **AI**: 테스트·구현 작성, 사이클 실행, 설계 트레이드오프 분석과 추천, Red/Green 예측(틀리면 정정)
- **사람**: 각 합의의 승인, 다음 작업 선택, 리팩토링 방향 조향, 스펙 결정, 요구사항 페이스 조절

## 핵심 규칙 6가지

1. **작업 유형에 맞는 첫 신호 하나** — 동작 변경은 Red 하나, 기존 동작 리팩토링은 특성화 테스트의 Green without Red 또는 기존 테스트의 기준 Green에서 시작한다
2. **실행 결과를 눈으로 확인** — Red·Green·기준선 판정은 반드시 실제 테스트 출력으로 증명한다
3. **유형 밖의 코드를 작성하지 않는다** — 동작 변경은 최소 구현만 작성하고, 기존 동작 리팩토링은 특성화 중 운영 코드를 바꾸지 않으며 리팩토링 중 새 행동을 추가하지 않는다
4. **AI는 테스트를 임의로 수정하지 않는다** — 테스트 변경은 반드시 사람 승인
5. **REFACTOR 사이클은 기준 Green에서만 시작한다** — 무엇을 왜 바꾸는지 설명하고 진행하며, 행동을 보존한다
6. **사람의 역할 = 조향** — 설계 결정이 갈리는 지점에서는 트레이드오프를 제시하고 사람이 결정한다

## 합의 단계 (코드 작성 전 필수, 계획 모드에서 시작)

합의 단계는 계획 모드에 진입해 시작한다. 코드 수정이 차단된 상태에서 컨텍스트를 쌓는다. 사이클이 계약의 이행이라면, 합의 단계는 계약을 쓰는 시간이다.

**합의 단계를 시작할 때 `references/agreement.md`를 반드시 읽는다** — 단계별 승인 절차(1차: 작업 유형 + 기능 정의·도메인 규칙 + API·테스트 작성 방식·구현 방향, 2차: 테스트 리스트, 3차: 진행 모드)와 합의 1~5의 작성 지침이 들어 있다. **진행 중이던 카타를 재개한 세션은 읽지 않는다** — spec·리스트 파일이 복원 지점이다.

### 사이클 중에도 유효한 합의 규칙

작업 유형 표 — 사이클 프로토콜이 이 용어를 참조한다:

| 유형 | 목적 | 기본적으로 기대하는 첫 신호 | 단계 완료 조건 |
|---|---|---|---|
| **동작 변경** | 없던 행동을 추가하거나 기존 행동을 변경 | 요구 행동을 표현한 테스트의 Red | 최소 구현으로 합의한 행동이 Green |
| **기존 동작 리팩토링** | 현재 행동을 특성화 테스트로 고정한 뒤 외부 행동을 바꾸지 않고 내부 구조를 변경 | 특성화 테스트의 Green without Red 또는 기존 테스트의 기준 Green | 필요한 현재 행동 문서화 + 구조적 완료 조건 충족 + 같은 테스트가 Green |

- 진행 중 목적이 바뀌면 현재 사이클을 멈추고 작업 유형을 다시 승인받는다. 승인 후 테스트 리스트·리팩토링 리스트·완료 조건을 단일 진실 공급원에 갱신한 뒤 재개한다
- 특성화 테스트가 예상과 달리 실패하면 원하는 동작에 맞춰 운영 코드를 곧바로 수정하지 않는다. 현재 행동과 기대 오라클 중 무엇이 틀렸는지 진단하고 스펙을 다시 합의한다
- 진행 중 추가되는 테스트 항목도 리스트 항목 형식을 따른다: **관찰 가능한 행동 하나 + 대표 입출력 예시**, 공개 API로 관찰 가능한 행동만, 문서화 후보는 미리 표시
- **TDD 스텝의 크기는 사람이 정한다.** AI는 기본적으로 최소 단위(사이클 하나)로 진행하고 더 작게 쪼개는 쪽을 추천하되, 사람은 연관 항목을 묶어 한 번에 승인할 수 있다 (예: 대칭 케이스 한 쌍, 같은 규칙의 연작, 즉시 통과가 예상되는 문서화 후보 일괄). 묶어서 진행할 때도 **Red는 한 번에 하나, 커밋은 사이클별**을 유지한다 — 묶이는 것은 승인이지 사이클이 아니다

### 합의 완료 후: 파일이 단일 진실 공급원

- 1~3차 승인이 모두 끝나면 첫 행동으로 세 파일을 만들고 합의 커밋을 남긴다:
  - `docs/tdd/<주제>-spec.md` — 합의문 중 **코드로도 스킬로도 복원 불가능한 것**: 작업 유형과 단계 완료 조건, 도메인 규칙·용어집, 확정 스펙과 스코프 밖 선언, 오라클 예제 표. 컨텍스트는 세션이 끝나면 휘발되므로 이 파일이 다음 세션의 스펙 복원 지점이다. 넣지 않는 것: API·구현 방향 스케치(코드가 진실), 테스트 표기 스타일(테스트 코드가 진실), 스킬 기본값 그대로인 진행 모드(스킬이 진실 — **기본값에서 벗어난 합의만** 기록한다)
  - `docs/tdd/<주제>-testlist.md` — 테스트 리스트만 (매 사이클 체크박스·문서화 표시 갱신)
  - `docs/tdd/<주제>-refactorlist.md` — 리팩토링 리스트 (시작 시 빈 목록). 보류된 리팩토링 후보를 보류 사유와 함께 기록하고, 완료 시 체크한다
- 이후 사이클마다 두 리스트 파일(testlist·refactorlist)의 체크박스와 문서화 테스트 표시를 갱신한다 — 대화가 아니라 파일이 단일 진실 공급원이다 (컨텍스트는 휘발된다)
- **완료된 항목도 파일에서 삭제하지 않는다** — 체크 표시(`[x]`)로 유지해 전체 이력을 보존한다. 남은 항목만 보여주는 것은 사이클 종료 보고의 표에서만이다
- 리스트는 살아있는 문서다 — 진행 중 케이스 추가·분할·수정이 가능하며, 변경은 사람이 승인한다 (아래 "진행 중 새 테스트 케이스 발견" 참조)
- **완주 후 커버리지 점검**: 리스트를 모두 완료하면 스위트의 구멍을 점검해 보고한다 (아래 "완주 후 커버리지 점검" 절의 관점과 처리 절차를 따른다)

## 사이클 프로토콜

**사이클은 두 종류다 — 한 사이클에 모자 하나만 쓴다** (Kent Beck의 두 모자: 행동 추가 vs 구조 개선):

| 사이클 | 시작 증거 | 작업 | 종료 증거 | 커밋 접두사 |
|---|---|---|---|---|
| **RED&GREEN** | 테스트의 Red (특성화는 Green without Red 선언) | 최소 구현 | 전체 Green | `RED&GREEN:` |
| **REFACTOR** | 전체 테스트 기준 Green (규칙 5) | 행동 보존 구조 변경 — 무엇을 왜 먼저 설명 | 테스트 무변경 전체 Green | `REFACTOR:` |

두 사이클 모두 같은 후처리로 끝난다: 리팩토링 리스트 최신화 → 두 리스트를 확인해 다음 작업 추천 → 사람이 선택.

### 🔴 Red (RED&GREEN 사이클)

- **동작 변경**에서는 실패하는 테스트 하나를 작성한다 (규칙 1)
- 반드시 실행해서 실패 출력을 공유하고 예상한 이유로 실패하는지 확인한다 (규칙 2)
- 컴파일용 껍데기가 필요하면 `UnsupportedOperationException`을 던지게 만들어 assertion 단계의 실패를 관찰한다
- **기존 동작 리팩토링의 특성화 단계**는 합의 단계에서 Green without Red를 예상했다고 선언한 경우 Red를 억지로 만들지 않는다
- **기존 동작 리팩토링의 리팩토링 단계**는 새 Red 대신 변경 전 기준 테스트 전체 Green을 출발 증거로 남긴다

### 🟢 Green (RED&GREEN 사이클)

- **동작 변경**은 구현 전에 방침을 한 줄로 선언한다: **`[최소]`** = 이 테스트만 통과하는 구현, **`[일반화]`** = 자연스러운 최소 구현이 미래 테스트까지 통과시키는 경우. `[일반화]`는 공짜로 통과될 리스트 항목을 명시하고 진행하며, 해당 항목은 통과 확인 후 문서화 테스트로 처리한다
- 동작 변경은 테스트를 통과시키는 최소 구현만 작성한다 (규칙 3). 하드코딩도 정답이다 — 다음 테스트가 일반화를 강제한다
- **기존 동작 리팩토링의 특성화 단계**는 운영 코드를 변경하지 않고 실제 통과 원인과 관찰된 행동을 문서화한다
- 구현 후 전체 테스트를 실행해 Green을 확인한다 (기존 테스트가 깨지지 않았는지 포함)

### 🔵 REFACTOR 사이클

**Refactor는 RED&GREEN의 부속 단계가 아니라 독립된 사이클이다** — 리팩토링 리스트에서 선택되어 시작한다. 예외적으로 방금 쓴 코드의 사소한 정리(이름, 자명한 중복)는 RED&GREEN 사이클 안에서 즉시 해도 되지만, 그 이상의 후보는 리스트에 올려 테스트 항목과 함께 저울질한다.

- 전체 테스트 기준 Green에서만 시작한다 (규칙 5). 무엇을 왜 바꾸는지 먼저 설명한다
- 행동 보존이 원칙 — **미래 요구사항 지식을 몰래 넣지 않는다**. 현재 테스트가 요구하지 않는 일반성을 도입하게 될 때는 반드시 언급한다
- 테스트는 수정하지 않는다 (규칙 4) — 리팩토링 후에도 테스트가 그대로 통과하는 것이 행동 보존의 증명이다
- 구조적 리팩토링(클래스 추출 등)은 사람이 먼저 제안할 수도 있다. AI는 찬반 트레이드오프를 모두 제시하고 추천 입장을 밝힌 뒤 조향에 따른다
- **다음 사이클이 바꿀 코드의 리팩토링은 보류가 기본이다.** 구조 변경은 다른 리팩토링의 비용을 바꾸므로, 보류 후보는 구조가 바뀔 때마다 재평가한다

### 사이클 공통 후처리 (두 유형 모두)

- 사이클 도중이라도, 발견했지만 지금 하지 않기로 한 리팩토링 후보는 **발견 즉시** 리팩토링 리스트 파일에 보류 사유와 함께 기록한다 — 후처리 시점까지 미루지 않는다
- **사이클이 끝날 때마다 리팩토링 리스트를 최신화한다** — ① 직전 사이클에서 변경·추가된 코드를 스멜 관점으로 분석해 새 후보를 기록하고(중복, 추상화 수준 불일치, 책임 배치, 매직 값 등 — 형식적 "없음" 선언이 아니라 실제 분석), ② 운영 코드 전체를 조망해 기존 보류 후보를 재평가한다(착수 / 계속 보류 / 불필요 제거 — 이번 변경으로 보류 사유가 해소되거나 비용이 달라진 항목)
- REFACTOR 사이클 뒤에는 **후속 정리도 점검한다**: 이동·추출로 참조 구조가 바뀌었으면 기존 멤버의 가시성(접근 제어자)을 최소로 좁힐 수 있는지, 더는 안 쓰는 멤버가 남지 않았는지 확인한다
- 최신화가 끝나면 두 리스트를 확인해 다음 작업을 추천하고 사람의 선택을 기다린다 (아래 "보고 및 다음 작업 선택")

### 보고 및 다음 작업 선택

사이클이 끝나면 아래를 보고하고, 다음 작업의 선택을 기다린다:

- 사이클 요약 — RED&GREEN이면 Red 증거 → Green 방침·구현 내용, REFACTOR면 무엇을 왜 바꿨는지 → 행동 보존 증거(테스트 무변경 Green). 공통으로 리스트 최신화 결과(신규/재평가)를 덧붙인다
- **남은 테스트 리스트와 리팩토링 리스트를 각각 별도의 표로 정리해 제시** (한 표에 섞지 않는다) — 두 표 모두 **남은 항목만** 보여준다. 리스트 파일도 함께 갱신한다. 표 형식은 아래 예시를 따른다:

  **📋 테스트 리스트 (남은 항목)**

  | # | 항목 | 예시 |
  |---|------|------|
  | 13 | 잘못된 명령 — 예외 + 위치 불변 | `FX` → `IllegalArgumentException`, `(0,0,N)` 유지 |

  **🔧 리팩토링 리스트 (남은 항목)**

  | # | 항목 | 보류 사유 / 비고 |
  |---|------|------|
  | R1 | `execute()` blocked 제어 흐름 정리 | 13번이 execute를 바꿀 예정 → 이후 재평가 |

- **다음 작업 추천**: **두 리스트를 모두 후보로 검토해** 추천과 근거(예상되는 Red/Green의 모양, 리팩토링이면 기대 효과)를 제시한다. 특히 보류 사유가 해소된 리팩토링(예: "N번 이후 재평가"의 N번이 완료됨)은 테스트 항목과 나란히 저울질하고, 테스트 항목이 곧 바꿀 코드의 리팩토링은 순서를 뒤로 미루는 근거를 명시한다. **다음 작업은 사람이 선택한다** — 순서 조향도 사람의 몫이다 (규칙 6)

### 커밋 규칙 (git 사용 시)

커밋은 항상 전체 테스트 Green 상태에서만 한다. 유형별 규칙:

- **합의 커밋**: 합의 단계 산출물(세 파일: spec·testlist·refactorlist) 저장 시 — `합의: <주제> 테스트 리스트와 합의 스펙`
- **RED&GREEN 사이클 커밋**: `RED&GREEN: <행동 요약>` — 본문에 Red 증거 한 줄을 남겨 히스토리만으로 TDD 이력을 증명하고, 테스트·리팩토링 리스트 파일 갱신을 같은 커밋에 포함한다
- **REFACTOR 사이클 커밋**: `REFACTOR: <무엇을 왜>` — 본문에 행동 보존 증거(테스트 무변경 전체 Green)를 남기고, 리스트 파일 갱신을 포함한다
- **테스트 커밋**: 구현 무변경의 테스트 스위트 변경(즉시 통과를 선언한 케이스 확장, 중복 통합·제거 등) — `테스트: <내용>`

## 완주 후 커버리지 점검

테스트 리스트의 전 항목이 Green이 되면, 단계 완료를 선언하기 전에 스위트의 구멍을 점검한다. **닫는 관문이 아니라 리스트를 한 번 더 열 수 있는 관문이다** — 발견이 있으면 항목이 추가되어 정상 절차로 처리되고, 없으면 완주를 선언한다.

**점검 관점 5가지**:

1. **숨은 단독 케이스**: 특수 규칙이 통합형 테스트 안에만 존재해 실패 신호가 뭉개지는 곳
2. **정의되지 않은 행동**: 합의에서 유보했거나 새로 드러난 미정의 입력·호출 순서 (예: 게임 종료 후 추가 조작, 자원 고갈)
3. **미검증 경계 조합**: 경계값이 단독으로는 검증됐지만 조합으로는 없는 곳
4. **규칙 교차 조합**: 서로 다른 규칙이 만나는 지점을 다루는 테스트가 있는가 (예: 딜러 턴 루프 × Ace 재계산 — 각각은 커버돼도 교차점은 빌 수 있다)
5. **구현 일반성의 강제 여부**: 루프·조건 같은 일반성이 테스트로 강제되는가 — 구현을 약화(while→if 등)해도 스위트가 통과하면 구멍이다 (뮤테이션 관점)

**발견 처리**: 각 발견을 선택지(테스트 추가 / 스코프 밖 선언 / 무시)와 추천으로 제시하고 **사람이 결정한다**. 테스트 추가가 승인되면 리스트 파일에 항목을 추가하고(발견 경위 명시) 문서화 테스트 또는 정상 Red 사이클로 처리한다. 스코프 밖 선언은 spec 파일에 근거와 함께 기록한다.

## 특수 패턴 처리

- **진행 중 새 테스트 케이스 발견**: 구현이나 논의 중 리스트에 없는 케이스·미정의 스펙을 발견하면, 발견 즉시 구현하지 않는다 — 선택지+추천으로 스펙을 합의한 뒤 항목을 **테스트 리스트 파일에 즉시 반영**하고(발견 경위 명시), 사이클로 다룬다
- **Green without Red** (새 테스트가 즉시 통과): 합의된 작업 유형이 **기존 동작 리팩토링의 특성화 단계**라면 기본적으로 기대하는 신호다. 추가 전에 "즉시 통과 예상"을 선언하고, 실제 통과 원인과 관찰된 행동을 기록해 회귀 방지용 **문서화 테스트로 유지**한다. 동작 변경에서 예상치 않게 즉시 통과했다면 과거 구현의 선행 일반화나 테스트 오라클 오류를 진단해 보고한다
- **예측과 정직한 정정**: Red/Green 모양을 예측했다가 틀리면 명시적으로 정정한다. 예측 실패는 숨길 일이 아니라 설계 이해도를 보정하는 데이터다
- **테스트 실행 증거**: 모든 Red/Green 판정은 실제 테스트 실행 출력으로 증명한다. "통과할 것이다"로 넘어가지 않는다
