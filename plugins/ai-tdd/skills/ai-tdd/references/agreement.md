# 합의 단계 상세 절차

카타/작업 **시작 시에만** 읽는 문서다. 사이클 진행 중에는 SKILL.md의 "사이클 중에도 유효한 합의 규칙"과 spec·리스트 파일이 기준이다.

## 단계별 승인

**승인은 한 번이 아니라 단계별로 받는다** — 도메인 합의가 확정되기 전에 리스트까지 한 뭉치로 승인받으면 승인의 해상도가 떨어진다:

1. **1차 승인 (계획 모드 승인)**: 합의 1(작업 유형), 합의 2(기능 정의·도메인 규칙), 합의 3(API·테스트 작성 방식·구현 방향 스케치)까지만 계획으로 제시하고 승인받는다. 테스트 리스트는 계획에 포함하지 않는다
2. **2차 승인**: 확정된 합의 2·3 위에서 합의 4(테스트 리스트)를 작성해 별도로 제시하고 승인받는다
3. **3차 승인**: 합의 5(진행 모드)를 별도로 논의해 확정한다

## 합의 1. 작업 유형 분류

코드나 테스트를 작성하기 전에 현재 작업 단계를 분류하고 사람의 승인을 받는다 (작업 유형 표는 SKILL.md에 있다):

- 사용자 의도, 현재 운영 코드의 행동, 기존 테스트 유무를 근거로 유형과 추천을 제시한다. AI가 작업 유형을 암묵적으로 정하지 않는다
- **기존 동작 리팩토링**은 ① 특성화 ② 리팩토링의 두 하위 단계로 진행한다. 특성화가 끝나고 기준 테스트 전체가 Green인 것을 확인하기 전에는 리팩토링을 시작하지 않는다
- 기존 테스트가 현재 행동을 충분히 보호한다면 근거와 실행 결과를 제시하고 사람의 승인을 받아 새 특성화 테스트 추가를 생략할 수 있다
- 한 요청에 두 유형이 섞이면 단계를 분리한다 (예: **기존 동작 리팩토링 → 동작 변경**). 한 시점에는 한 유형만 진행한다
- 각 단계마다 **작업 유형과 하위 단계, 예상 Red/Green 모양, 허용되는 운영 코드 변경, 단계 완료 조건**을 합의문에 기록한다
- 특성화 완료 후 리팩토링을 시작하기 전에 구조적 완료 조건을 리팩토링 리스트에 기록하고 사람의 승인을 받는다

## 합의 2. 기능 정의와 도메인 규칙 정리

- 만들 기능이 무엇인지 한두 문장으로 정의하고, 구현에 필요한 도메인 규칙만 짧게 요약한다
- 용어집을 정리한다 — 여기서 확정한 어휘가 코드·테스트의 이름이 된다
- 대표 시나리오의 기대값을 손으로 계산해 예제 표로 정리한다 (테스트 오라클 검증 — 기대값이 틀린 테스트는 TDD 전체를 오염시킨다)
- **암묵 스펙을 표면화한다**: 테스트 리스트의 모양 자체가 스펙 결정을 내포할 수 있다 (예: "프레임 하나짜리 테스트 허용" = "진행 중 게임의 score() 조회 허용"이라는 스펙 채택). 이런 결정은 선택지+추천으로 제시하고 합의한다
- 정의되지 않은 행동(예: 게임 완료 후 추가 투구)을 발견하면 리스트에 넣을지 스코프 밖으로 선언할지 결정한다

## 합의 3. 공개 API 합의, 테스트 작성 방식, 구현 방향 스케치

- 테스트가 사용할 인터페이스를 먼저 합의한다 (예: `game.roll(pins)` / `game.score()`). 도메인 개념(예: 프레임)을 API에 노출할지 내부에 둘지도 이 시점에 언급한다
- **테스트를 어떻게 작성할지 합의한다** — 진행 중 즉흥 결정하게 되면 전환 비용(테스트 커밋)이 쌓이므로 여기서 정한다:
  - **테스트 고도**: 모든 행동을 최상위 API 시나리오로 관찰할지(내부 구조 전체가 리팩토링 자유 영역이 됨), 도메인 객체별 단위로 관찰할지(실패 국소화가 정밀), 혼합할지 — 트레이드오프+추천을 제시하고 사람이 결정한다. **계약 밖으로 둘 내부 구조를 명시한다** (예: "Hand·Deck의 존재와 형태는 계약 밖"). 진행 중 상위 Red의 원인이 애매해지면 하위 테스트 추가를 그때 승인받는 탈출구를 둔다
  - **표기 스타일**: assertion 라이브러리, 테스트 명명 규칙(도메인 언어 사용 여부)
  - **셋업 방침**: 픽스처·헬퍼의 형태(예: 덱 주입 헬퍼). 처음부터 완벽할 필요는 없고, 보일러플레이트가 반복되면 리팩토링 리스트에 올려 승인 후 도입한다
- **구현 방향을 가볍게 스케치한다**: 유력한 내부 구조 후보들을 트레이드오프와 함께 한두 문장씩 논의한다 (예: 투구 리스트 + 인덱스 순회 vs 프레임 객체 + 즉시 정산). 단, 스케치는 **약속이 아니라 가설이다** — 테스트가 다른 방향을 강제하면 폐기하고, 스케치를 근거로 선행 구현하지 않는다 (규칙 1은 그대로 유효)
- API와 구현 방향이 먼저 정리되어야 다음 단계의 리스트 품질 점검(간격, 공짜 통과 예측)이 정확해진다

## 합의 4. 테스트 리스트 작성

각 항목은 **관찰 가능한 행동 하나 + 대표 입출력 예시**로 쓴다 (예: "스페어 보너스 — 5,5 / 3 → 16"). 예시가 있어야 사이클 중 "공짜 통과" 판단과 Red 예측이 정확해진다. **항목은 공개 API로 관찰 가능한 행동만 다룬다** — 합의 3에서 스케치한 내부 구조를 검증하는 항목을 넣지 않는다 (리스트는 행동의 계약이지 설계의 계약이 아니다).

리스트 품질 점검 4가지:

1. **크기**: 첫 테스트가 전체 시나리오를 요구하면 더 작은 계단으로 분할한다 (전체 게임 → 프레임 하나)
2. **커버리지**: 특수 규칙마다 ① 단독 케이스 ② 조합·연속 케이스 ③ 경계 케이스가 각각 있는지 점검한다. 통합형 테스트(예: 퍼펙트 게임)에만 의존하면 단독 케이스가 그 안에 숨어 실패 신호가 뭉개진다
3. **간격**: 인접 항목 사이의 구현 점프가 너무 크면 중간 계단을 추가한다 (단일 스트라이크 → 퍼펙트 게임 사이에 연속 스트라이크)
4. **정렬**: 뼈대(단순 경로) → 특수 규칙 → 조합 → 경계·통합 순으로 배치한다

**문서화 테스트 후보를 미리 표시한다**: 앞 항목의 자연스러운 구현이 통과시킬 것으로 예상되는 항목(예: 거터 게임)은 리스트에 표시해둔다. 예측이 맞으면 문서화 테스트로 추가하고, 틀리면 정상 사이클로 진행한다 — 어느 쪽이든 데이터가 된다.

## 합의 5. 진행 모드 합의

기본은 사이클 단위 승인 — 사이클이 끝날 때마다 요약을 보고하고 다음 작업 선택을 기다린다. git 사용 여부와 커밋 규칙 적용도 이때 확정한다. 스킬 기본값에서 벗어난 합의만 spec 파일에 기록한다.
