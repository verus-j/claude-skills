# Code Smell Detector

두 개의 카탈로그를 기준으로 코드에서 리팩토링이 필요한 신호를 찾아내고, 각 스멜에 맞는 표준 리팩토링 기법을 처방하는 스킬이다.

- **Fowler & Beck 카탈로그** (Refactoring 3장, 22개 스멜): 클래스·메서드 **구조**의 리팩토링 신호. 처방이 표준 리팩토링 기법명으로 연결된다.
- **Clean Code 휴리스틱** (Martin, Clean Code 17장, C/E/F/G/J/N/T 계열): 주석, 네이밍, 함수 설계, 테스트, 빌드 환경까지 포함한 더 넓은 **가독성/설계** 휴리스틱.

핵심 철학: 스멜은 "버그"가 아니라 **리팩토링 가능성을 시사하는 신호**다. 정밀한 지표(줄 수 몇 줄 이상이면 무조건 문제)보다 맥락을 읽은 판단이 우선이다. 지표는 참고용으로만 쓰고, "이 구조가 앞으로의 변경을 어렵게 만드는가?"를 기준으로 판단한다.

## 워크플로우

### 1. 코드 파악
- 분석 대상 파일(들)을 읽는다. 디렉토리라면 소스 파일 목록을 먼저 확인하고 주요 파일부터 읽는다.
- 코드가 무엇을 하는지 먼저 이해한다. 도메인을 모르고 스멜만 찾으면 오탐이 많아진다.
- 클래스/함수 간 관계(누가 누구를 호출하고, 어떤 데이터가 어디로 흐르는지)를 파악한다. Feature Envy, Shotgun Surgery, Message Chains 같은 스멜은 단일 함수만 봐서는 안 보인다.

### 2. 스멜 스캔
`references/smell-catalog.md`와 `references/clean-code-heuristics.md`를 읽고 감지 신호를 기준으로 코드를 훑는다. 효율적인 스캔 순서:

1. **구조 스멜부터**: Duplicated Code, Long Method, Large Class, Long Parameter List — 눈에 가장 잘 띄고 빈도가 높다.
2. **변경 축 스멜**: Divergent Change, Shotgun Surgery — "이 코드에 새 요구사항이 오면 어디를 고쳐야 하나?"를 상상하며 찾는다.
3. **책임 배치 스멜**: Feature Envy, Data Class, Inappropriate Intimacy, Middle Man, G6 잘못된 추상화 수준, G13 인위적 결합, G17 잘못 놓인 책임 — 데이터·행동·상수가 있어야 할 곳에 있지 않은 경우를 찾는다.
4. **타입/조건 스멜**: Switch Statements, Primitive Obsession, Data Clumps, G28 조건식 캡슐화, G29 부정형 조건.
5. **표현/의도 스멜**: 주석(C1~C5), 이름(N1~N7), G16 가려진 의도, G19 설명 변수, G20 함수 이름, G25 매직 넘버 — 코드가 자기 의도를 말하지 못하는 곳을 찾는다.
6. **정밀성/안전성 스멜**: G2 미구현된 당연한 동작, G3 경계 동작 오류, G4 안전장치 무력화, G26 부정밀한 결정, G31 숨은 시간적 결합, Dead Code(G9/F4).
7. **테스트/환경 스멜**: 테스트 코드가 분석 대상에 있으면 T1~T9, 빌드 스크립트가 있으면 E1~E2. 소스 파일만 받았으면 이 계열은 건너뛴다.
8. **기타**: 나머지 스멜은 위 과정에서 자연스럽게 드러나는 경우가 많다.

주의사항:
- 같은 근본 원인이 여러 스멜로 나타날 수 있다(예: Large Class 안에 Long Method 다수). 이 경우 근본 원인을 대표 스멜로 잡고, 나머지는 그 항목 안에서 함께 언급한다. 같은 문제를 스멜 개수 늘리기로 중복 보고하지 않는다.
- **두 카탈로그가 겹치는 항목은 하나로만 보고한다.** Fowler 쪽 명칭을 대표로 쓰고 Clean Code ID를 병기한다: `Duplicated Code (G5)`, `Long Parameter List (F1)`, `Feature Envy (G14)`, `Switch Statements (G23)`, `Message Chains (G36)`, `Long Method (G30)` 등. Clean Code에만 있는 항목은 ID + 명칭으로 보고한다: `G16 Obscured Intent`, `N7 Names Should Describe Side-Effects`.
- 스멜이 아닌 것을 억지로 찾지 않는다. 깨끗한 코드에는 "특이사항 없음"이라고 말할 수 있어야 신뢰받는 리포트다.
- Refused Bequest처럼 원저자들도 "열에 아홉은 무시해도 된다"고 한 약한 스멜은 실제로 혼란을 일으키는 경우에만 보고한다. J1(와일드카드 import)은 현대 컨벤션과 충돌하므로 기본적으로 보고하지 않는다.
- N5(스코프 길이에 비례하는 이름)를 기억한다: 짧은 루프의 `i`는 스멜이 아니다. G25의 예외(자명한 상수)도 마찬가지 — 휴리스틱을 기계적으로 적용하지 않는다.

### 3. 심각도 판정
각 스멜에 심각도를 매긴다. 기준은 "코드가 못생겼는가"가 아니라 **변경 비용과 버그 위험**이다.

| 심각도 | 기준 | 예시 |
|---|---|---|
| 🔴 High | 지금 당장 변경을 어렵게 하거나 버그를 유발 중. 수정 시 누락 위험이 큼 | 3곳 이상 중복된 코드, 여러 클래스에 흩어진 switch문, 수백 줄 God Class, 안전장치 무력화(G4), 경계 동작 오류(G3), float 통화 같은 부정밀 결정(G26) |
| 🟡 Medium | 당장 아프진 않지만 다음 기능 추가 때 비용이 커질 구조 | 긴 파라미터 목록, Feature Envy, Data Clumps, 숨은 시간적 결합(G31), 오해를 부르는 이름(G20/N4/N7), 매직 넘버(G25) |
| 🟢 Low | 알고 있으면 좋은 수준. 관련 코드를 건드릴 때 함께 정리하면 됨 | 약한 Refused Bequest, 사소한 Lazy Class, 주석으로 때운 코드, 중복 주석(C3), 주석 처리된 코드(C5), 잡동사니(G12) |

### 4. 리포트 작성
핵심 구성: **표 하나로 전체를 한눈에, 상세 설명은 High만 짧게.** 사용자가 대화에서 쓰는 언어로 작성한다.

```markdown
# 코드 스멜 리포트 — (파일/범위)

**총평**: 한두 문장. 코드의 전반적 상태와 최우선 개선 포인트.

| # | 심각도 | 스멜 | 위치 | 처방 |
|---|---|---|---|---|
| 1 | 🔴 | Duplicated Code (G5) | process_order ↔ estimate_shipping | Extract Method로 통일 |
| 2 | 🔴 | Long Method (G30) | process_order (70줄) | 단계별 Extract Method |
| 3 | 🟡 | Data Clumps | 주소 4개 필드가 몰려다님 | Extract Class → Address |
| 4 | 🟡 | G20 Function Names Should Say What They Do | save()가 검증+발송까지 함 | validateAndSend로 분리·개명 |
| 5 | 🟢 | Primitive Obsession | 금액을 int로 처리 | 필요해지면 Money 도입 |
| 6 | 🟢 | C5 Commented-Out Code | order.py:112-131 | 삭제 |

## 🔴 자세히
**1. Duplicated Code** — 배송비 계산이 두 메서드에 완전히 동일하게 존재.
기준 금액이 바뀌면 두 곳을 고쳐야 하고, 한 곳만 고치면 실제 버그. → 한 메서드로 추출해 양쪽에서 호출.

**2. Long Method** — ...(항목당 2~3문장: 증상 + 왜 문제 + 처방 방향)
```

리포트 작성 원칙:
- **표가 리포트의 중심이다.** 표만 읽어도 전체 상황이 파악되어야 한다. 표의 처방 칸은 기법명 위주로 5단어 이내.
- **상세 설명은 🔴 High만.** Medium/Low는 표의 한 줄로 충분하다. 사용자가 특정 항목을 더 물으면 그때 설명한다.
- **상세 항목은 2~3문장.** 증상 → 왜 문제 → 처방 방향을 압축한다. 소제목·굵은 라벨 남발 금지.
- **위치는 구체적으로**: "이 클래스가 좀 큽니다"가 아니라 `process_order (order.py:28-96, 70줄)`처럼.
- **리팩토링 기법명은 카탈로그의 표준 명칭 사용**: Extract Method, Move Method 등. 사용자가 Fowler 책에서 바로 찾을 수 있게. Clean Code 항목은 ID(G16, N4 등)를 붙여 책 17장에서 바로 찾을 수 있게.
- **작업 순서를 추천하지 않는다.** 리포트는 발견된 작업 목록까지만이다. 어떤 스멜부터 손댈지는 사용자가 정한다. "1번부터 하세요", "권장 순서" 같은 제안을 넣지 않는다. 단, 리팩토링 간 사실적 의존 관계(예: "Address 추출 시 5·6번이 함께 해소됨")는 판단에 필요한 정보이므로 해당 항목의 처방 안에 한 줄로 적어도 된다.
- 스멜이 없으면 표 없이 "특이사항 없음" + 잘된 점 한두 문장으로 끝낸다.

### 5. 출력 방식: 인라인 vs md 파일
리포트 규모에 따라 결정한다:

- **작은 분석** (파일 1~2개, 스멜 ~7건 이하): 대화에 인라인으로. 파일을 만들면 오히려 번거롭다.
- **큰 분석** (여러 파일, 스멜 8건 이상, 또는 사용자가 저장/공유를 언급): 전체 리포트를 md 파일로 저장하고, 대화에는 **총평 + High 항목 표만** 보여준 뒤 파일을 전달한다. 사용자가 리포트를 리팩토링 체크리스트로 재사용하거나 팀에 공유할 수 있다.
- 파일로 저장했다면 이후 대화에서 항목 번호로 참조한다. 사용자가 "4번부터 하자"라고 하면 리포트를 다시 설명하지 않고 바로 이어간다.

### 6. (요청 시) 리팩토링 예시
사용자가 원하면 High 항목 1~2개에 대해 before/after 코드를 보여준다. 요청 없으면 리포트까지만.
